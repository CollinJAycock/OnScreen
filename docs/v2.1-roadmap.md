# v2.1 Release Roadmap

Mostly closes the gaps left by v2.0's scoping decisions, plus the bug-shape fixes carried out of the beta soak. The big-ticket item — native clients — is the size of all other tracks combined and may warrant its own release line if shipping cadence is a concern.

Anchored in [v2-roadmap.md](v2-roadmap.md)'s deferred list and [RELEASE.md](../RELEASE.md). Items added here are either explicitly tagged in those docs or surfaced during the 2026-04 beta soak.

---

## Track A — Bug-shape fixes carried out of v2.0

The bag the beta soak left under the door. None of these blocked v2.0, but each is a known quality issue worth clearing before adding new feature surface.

| Item | Complexity | Notes |
|---|---|---|
| Async OCR endpoint | S | `POST /items/{id}/subtitles/ocr` is request-bound and 524s behind any reverse proxy with sub-multi-minute response timeout (Cloudflare Tunnel free tier = 100s). Convert to job-queued: POST returns 202 + job_id, `GET /subtitles/ocr/{jobId}` polls status, OCR runs via `context.Background()` in a server-lifetime goroutine. Watch-page UI polls instead of awaiting. The `ocr_subtitles` scheduler task already runs out-of-band correctly — same pattern, exposed per stream. **Highest impact-per-line item on this list — actively biting beta users.** |
| Valkey-backed SAML request tracker | S | The in-memory tracker that ships in v2.0 (`memorySAMLRequestTracker`) is single-instance only. Multi-instance OnScreen behind a load balancer would lose tracked requests if the AuthnRequest hits one instance and the ACS hits another. Drop-in replacement implementing the same `samlsp.RequestTracker` interface backed by the existing Valkey client (~50 LOC). Only matters when someone runs HA OnScreen — not urgent unless that becomes a real deployment shape. **v2.1 status: shipped.** `valkeySAMLRequestTracker` mirrors the memory contract (10-minute TTL via Valkey SETEX, RelayState-keyed lookup, single-use semantics), and `SAMLHandler.WithRequestTracker` lets cmd/server inject it. Used unconditionally in the prod wiring so dev and HA share the same code path. |
| `settings.test.ts` SMTP-render fixture | XS | Pre-existing Vitest failure on `main` predating v2.0. Test-fixture issue, not a code regression. Worth fixing for green CI signal. |

## Track B — New media types

Important nuance discovered during v2.1 planning: **the schema + scanner work for music videos, audiobooks, and podcasts already shipped in v2.0** (migrations 00055, 00056, 00057; scanners with 14 tests across the three). The work was done but not announced in the v2.0 release notes because the frontend never got dedicated landing pages, chapter UI, or feed-driven flows. v2.1 closes the frontend gap and adds the two genuinely-new types (home videos, books).

| Item | v2.0 status | Remaining for v2.1 | Complexity |
|---|---|---|---|
| Music videos | Schema ✓, scanner ✓, artist-page renders the row ✓ | Library-page typed shelf (`library.type='music'` shows a Music Videos row alongside albums); generic `/libraries/{id}` doesn't yet special-case the music_video child type. | XS |
| Audiobooks | Schema ✓, scanner ✓ (`audiobook` library + item type), 5 scanner tests | Author/series hierarchy (deferred per migration 00056 comment), chapter scrubber UI driven by ffprobe markers, library page audiobook-grid layout, resume snaps to chapter boundary not seconds. | M |
| Podcasts (local files only) | Schema ✓, scanner ✓ (`podcast` + `podcast_episode` types), 3 scanner tests | Library page podcast-show grid, episode list per show, no RSS auto-download (out of scope per project memo). | S |
| Home videos | Nothing yet | New `home_video` item type for personal footage (vacations, family events, ranch cams). No external metadata agent — TMDB doesn't have your kid's birthday party. Date sourced from file mtime + ffprobe `creation_time` with EXIF fallback. Primary library sort is date-taken (like photos), not alphabetic title. UI shows a date-grouped grid rather than the cover-art-heavy movie shelf. Event folders (`Home Videos/Yellowstone 2024/`) auto-promote to collections. Reuses existing video transcode + trickplay pipelines unchanged. | S |
| Books / comics | Nothing yet | CBR/CBZ/EPUB parsing (new deps: `archive/zip`, `archive/rar` via golang-rar, `epub` parser). Separate reader UI is the dominant cost — pagination, full-text search, bookmarks, reading progress. Smallest user overlap on this track, but completes the "all-in-one library" story. | L |

**Dependency order for the remaining work:** music videos shelf (XS, validates the typed-library-page pattern) → podcasts UI (reuses the same pattern) → audiobooks chapter UI (genuinely new — no chapter scrubber exists in v2.0) → home videos (date-grouped grid is the new pattern) → books/comics (the reader UI is the dominant cost).

## Track F — Discovery and recommendations

The "what should I watch next?" surface area. v2.0's home hub has Continue Watching + Recently Added per library, but no actual recommendation layer. This track adds the discovery rows that turn a library full of media into a feed worth opening.

| Item | Complexity | Notes |
|---|---|---|
| Smart playlists (rule-based) | M | User-defined logical rules — "all 4K HDR movies from 2020+", "songs not played in 6 months", "TV episodes I've started but not finished". JSON rule schema, per-rule evaluator that compiles to a parameterised query, admin UI for rule editing (probably condition rows + AND/OR groupings, mirroring iTunes / Plex). Live-evaluated at query time so newly-imported items appear without a rebuild. Reuses the existing `playlists` schema with a `rules JSONB` column instead of an explicit item list. **Highest leverage on this track — many users want this independent of any ML.** |
| pgvector item-embedding pipeline | M | Scaffolding shipped in v1 (pgvector extension optional in `00001_init.sql`, embedding column reserved on `media_items`); the actual embedding column, generation pipeline, and ivfflat index are still to land. Embed each item from its TMDB/MusicBrainz description via a local model (sentence-transformers via Python sidecar, or an Ollama-served embedding model); index with `ivfflat` on cosine distance. Backfills the existing library on first run, then incrementally on scan. **Foundation for the next two items — neither works without this.** |
| Trending row | S | Aggregate `watch_events` over a rolling window (7d / 30d), top-N by play count, exposed as a hub row. No personalization — same list for every user. Reuses existing schema; just a query + UI surface. Cheap to ship. |
| "Because you watched X" | M | Per-user recommendations: take the user's recently-completed items, ANN-search pgvector for nearest neighbours, exclude items already watched, expose as a hub row tagged with the seed item ("Because you watched *Apocalypse Now*"). Depends on the embedding pipeline above. |

**Dependency order:** smart playlists (independent, ship first) → pgvector embeddings (foundation) → trending (independent, can parallelise with embeddings) → because-you-watched-X (depends on embeddings).

## Track H — Streaming format extensions

v2.0 ships HLS with fMP4 segments (already CMAF-compatible). DASH adds a parallel manifest format alongside the same segment ladder, broadening client compatibility for the non-Apple side of the ecosystem.

| Item | Complexity | Notes |
|---|---|---|
| MPEG-DASH streaming | M | Generate `.mpd` manifests alongside the existing `.m3u8` for the same fMP4 segment ladder. Most of the cost is the manifest writer — segment encoding is unchanged because v2.0 already emits CMAF-compatible fMP4 (one ladder, two manifests). Frontend swaps `hls.js` for `shaka-player` (handles both HLS and DASH), or runs a parallel `dash.js`; pick at session start based on `Accept`/UA. Test matrix is the actual work — Chromecast, AndroidTV, Samsung Tizen, LG webOS, Roku each prefer different things. **Most useful for the smart-TV side; browser playback works on either format.** **v2.1 status:** server-side shipped — `BuildMPD` writer + `GET /api/v1/transcode/sessions/{sid}/manifest.mpd` endpoint serve fMP4 sessions (HEVC today, AV1 once that path lands). Token gating mirrors the HLS playlist; MPEG-TS sessions return 415 with a hint to fall back to `playlist.m3u8`. Frontend `shaka-player` integration + smart-TV test matrix are the next deliverable. |

## Track G — Per-user policy and parental controls

The `LibraryAccessChecker` hook is wired end-to-end across v2.0's handlers (items, collections, playlists, photo-albums, search, history, favorites, hub, plus the `/users/{id}/libraries` admin endpoints), and the per-user grant table is functional — but the **policy layer that drives it is still effectively global by default**. Most installs never opt into per-user grants because there's no library-level "private" flag and no good default-deny onboarding flow. v2.1 closes the loop.

| Item | Complexity | Notes |
|---|---|---|
| Library-level visibility flag | S | Add `is_private BOOLEAN DEFAULT false` to `libraries`. Public libraries are visible to every authenticated user (current behavior). Private libraries require an explicit grant in `library_access`. Lets admins mark a single "Family Movies" library as public while keeping "4K Movies" private behind grants — without flipping the default for the whole install. |
| Default-grant template | S | Admin setting: "When a new non-admin user is created, grant access to libraries [...]." Picks up new accounts that come from OIDC/SAML/LDAP auto-provisioning so they don't land on a barren home page. Today the default for new non-admin users is zero libraries — strict but startling. |
| Profile-level library visibility | M | Managed profiles (kid mode etc.) inherit the parent user's grants today. v2.1 lets admins narrow per-profile — kid profile sees `Family Movies` only even though the parent's account has access to `4K Movies` too. Couples to the parental-rating-ceiling work that shipped in v2.0. |
| Content-rating filter within an allowed library | M | Per-profile (and per-user) ceiling that hides items above the configured rating *inside* a library the user can otherwise see. Half-shipped in v2.0 (parental-rating-ceiling field exists on managed profiles); v2.1 wires it into hub queries, search results, and the items endpoint as a query-time filter. |
| Audit "view as" tool | S | Admin UI that renders the home hub / library page exactly as a target user would see it — verifies the policy stack end-to-end without having to log in as the user. Feeds the manual test plan's "managed-profile content-rating" row. |

**Dependency order:** library-private flag (foundation — every other item assumes the public/private split exists) → default-grant template → profile-level visibility → content-rating filter → audit "view as" tool.

## Track E — The native client phase

The biggest scope item on this list, and the one that unlocks the v2-roadmap's "audiophile pillar" pitch (browser audio pipelines can't deliver bit-perfect; native clients can).

| Item | Complexity | Notes |
|---|---|---|
| Hardware bit-perfect playback | M | Couples to native clients — the browser audio API doesn't expose WASAPI exclusive / CoreAudio HOG / ALSA hw access. The server side is already correct (FLAC direct stream, no transcode); the missing piece is a client that can take the byte stream straight to the audio driver. |
| Native client apps | XL | Windows, macOS, Linux, iOS, Android, TV. Project memory tags this as the post-v2 phase. Tooling decision needed up front: Tauri + Svelte (reuses the existing web frontend), or platform-native per-platform. Per-platform is more code; Tauri reuses 80% of the work but loses some platform integration polish. |
| OnScreen device sync between clients | M | Resume position, library state, watch history sync across the user's own devices — Plex's "On Deck" parity. Server-side is mostly there; client-side needs the sync protocol. |

**Dependency order:** tooling decision (Tauri vs native) → desktop client first (largest leverage, already-shipped backend) → mobile → TV.

## Open Questions

- **Cadence:** ship v2.1 as one big release with all six tracks, or split? Reasonable splits:
  - v2.1 = Track A + D + F (~4 weeks — clears beta debt + lands smart playlists & trending)
  - v2.2 = Track B + C (new media types + Tidal)
  - v2.3 = Track E (native clients)
  - This trades release count for shorter feedback loops on each batch.
- **Tidal/Qobuz:** still the right v2.1 priority? Audiophile users will rate it differently from convenience users. The native client phase delivers more bit-perfect value than streaming integration alone.
- **Native client tooling:** Tauri or platform-native? The Tauri spike is cheap; the platform-native investment is multi-quarter.
- **Audiobooks chapter resume protocol:** does it match Plex's behavior (resume to chapter boundary) or our own (resume to second)? Matters for users migrating from Plex with multi-thousand-hour libraries.

## Items explicitly NOT in v2.1

- **Podcast RSS auto-fetch** — out of scope per project memo. OnScreen doesn't download content; local podcast files work via the regular library scanner.
- **Live TV theme music / EXIF-only book metadata** — earlier scoping calls.
- **Inbound MCP server** — plugin direction is outbound-only by design (security).
