# v2.1 Release Roadmap

Mostly closes the gaps left by v2.0's scoping decisions, plus the bug-shape fixes carried out of the beta soak. The big-ticket item — native clients — is the size of all other tracks combined and may warrant its own release line if shipping cadence is a concern.

Anchored in [v2-roadmap.md](v2-roadmap.md)'s deferred list and [RELEASE.md](../RELEASE.md). Items added here are either explicitly tagged in those docs or surfaced during the 2026-04 beta soak.

---

## Track A — Bug-shape fixes carried out of v2.0

The bag the beta soak left under the door. None of these blocked v2.0, but each is a known quality issue worth clearing before adding new feature surface.

| Item | Complexity | Notes |
|---|---|---|
| Async OCR endpoint | S | `POST /items/{id}/subtitles/ocr` is request-bound and 524s behind any reverse proxy with sub-multi-minute response timeout (Cloudflare Tunnel free tier = 100s). Convert to job-queued: POST returns 202 + job_id, `GET /subtitles/ocr/{jobId}` polls status, OCR runs via `context.Background()` in a server-lifetime goroutine. Watch-page UI polls instead of awaiting. The `ocr_subtitles` scheduler task already runs out-of-band correctly — same pattern, exposed per stream. **Highest impact-per-line item on this list — actively biting beta users.** |
| Valkey-backed SAML request tracker | S | The in-memory tracker that ships in v2.0 (`memorySAMLRequestTracker`) is single-instance only. Multi-instance OnScreen behind a load balancer would lose tracked requests if the AuthnRequest hits one instance and the ACS hits another. Drop-in replacement implementing the same `samlsp.RequestTracker` interface backed by the existing Valkey client (~50 LOC). Only matters when someone runs HA OnScreen — not urgent unless that becomes a real deployment shape. |
| `settings.test.ts` SMTP-render fixture | XS | Pre-existing Vitest failure on `main` predating v2.0. Test-fixture issue, not a code regression. Worth fixing for green CI signal. |

## Track B — New media types

Picks up the v2-roadmap Track A items that didn't make v2.0's cut.

| Item | Complexity | Notes |
|---|---|---|
| Audiobooks | L | New `audiobook` item type + `audiobook_chapters` table. Reuse music scanner for tag parsing (m4b / mp3). Chaptered resume state (resume should land on the chapter boundary, not the second). No open Audible API — Google Books + local chapter markers is the workable agent path. |
| Music videos | S | New type; mostly a scanner branch that routes files with music-video tags or a `Music Videos/` folder convention to a separate type. Reuses the existing video transcode pipeline. **Smallest item in the track — ship first to validate the new-type wiring.** |
| Books / comics | L | CBR/CBZ/EPUB parsing (new deps: `archive/zip`, `archive/rar` via golang-rar, `epub` parser). Separate reader UI is the dominant cost — pagination, full-text search, bookmarks, reading progress. Smallest user overlap of any item on this list, but completes the "all-in-one library" story. |

**Dependency order:** music videos (smallest, validates new-type plumbing) → audiobooks → books/comics.

| Item | Complexity | Notes |

| Tidal / Qobuz full streaming | XL | v2.0 has these scoped to "metadata supplement only" — use Tidal's metadata as another agent source. v2.1 adds OAuth bind, library import, streaming passthrough, ReplayGain alignment with the local FLAC pipeline, and the licensing legwork. ## Track D — Quality / regression coverage

The chores that stop the next beta soak from finding the same kinds of holes.

| Item | Complexity | Notes |
|---|---|---|
| Playwright auth-providers spec | S | Wire the `docker/docker-compose.idp.yml` stack into a Playwright `globalSetup`, run OIDC + SAML + LDAP rounds against a local OnScreen instance, tear down. The IdP stack is already documented in `docs/auth-test-stack.md` — this is the missing fourth file under `web/tests/e2e/`. Locks the SAML work behind a regression guard so the four-layer signing/tracker fix can't silently break. |
| Dependabot triage | S | 10 PRs queued at v2.0 tag time (cuda 13.2.1 runtime, node 25 alpine, vite 8, typescript 6, vite-plugin-svelte 7, golangci-lint 9, setup-buildx 4, setup-node 6, npm aggregate, Go aggregate). Triage on `main` post-tag. Most are mechanical; the cuda runtime bump may need an FFmpeg-image rebuild check. |
| `gh` CLI in dev environment | XS | Not having it meant the GitHub release form had to be filled in by hand. Install on the dev box / document as a project prereq so future releases are one command. |

## Track E — The native client phase

The biggest scope item on this list, and the one that unlocks the v2-roadmap's "audiophile pillar" pitch (browser audio pipelines can't deliver bit-perfect; native clients can).

| Item | Complexity | Notes |
|---|---|---|
| Hardware bit-perfect playback | M | Couples to native clients — the browser audio API doesn't expose WASAPI exclusive / CoreAudio HOG / ALSA hw access. The server side is already correct (FLAC direct stream, no transcode); the missing piece is a client that can take the byte stream straight to the audio driver. |
| Native client apps | XL | Windows, macOS, Linux, iOS, Android, TV. Project memory tags this as the post-v2 phase. Tooling decision needed up front: Tauri + Svelte (reuses the existing web frontend), or platform-native per-platform. Per-platform is more code; Tauri reuses 80% of the work but loses some platform integration polish. |
| OnScreen device sync between clients | M | Resume position, library state, watch history sync across the user's own devices — Plex's "On Deck" parity. Server-side is mostly there; client-side needs the sync protocol. |

**Dependency order:** tooling decision (Tauri vs native) → desktop client first (largest leverage, already-shipped backend) → mobile → TV.

## Open Questions

- **Cadence:** ship v2.1 as one big release with all five tracks, or split? Reasonable splits:
  - v2.1 = Track A + D (~2 weeks of work, ships fast — clears beta debt)
  - v2.2 = Track B + C (new media types + Tidal)
  - v2.3 = Track E (native clients)
  - This trades release count for shorter feedback loops on each batch.
- **Tidal/Qobuz:** still the right v2.1 priority? Audiophile users will rate it differently from convenience users. The native client phase delivers more bit-perfect value than streaming integration alone.
- **Native client tooling:** Tauri or platform-native? The Tauri spike is cheap; the platform-native investment is multi-quarter.
- **Audiobooks chapter resume protocol:** does it match Plex's behavior (resume to chapter boundary) or our own (resume to second)? Matters for users migrating from Plex with multi-thousand-hour libraries.

---

## Items explicitly NOT in v2.1

- **Podcast RSS auto-fetch** — out of scope per project memo. OnScreen doesn't download content; local podcast files work via the regular library scanner.
- **Live TV theme music / EXIF-only book metadata** — earlier scoping calls.
- **Inbound MCP server** — plugin direction is outbound-only by design (security).
