# v2.2 Release Roadmap

Seed for the next minor. v2.1 is tagged ([CHANGELOG.md](../CHANGELOG.md)) and shipping; v2.1.1 is the imminent dot-release for the Play-Store-readiness work and self-service account deletion. v2.2 is where the next batch of feature tracks lands.

This doc starts with two media-type tracks — **anime** and **manga / webtoons** — that emerged from the post-v2.1 product review. More tracks will accumulate here as scope is decided.

Anchor: [comparison-matrix.md](comparison-matrix.md) for what we lead / trail. [v2.1-roadmap.md](v2.1-roadmap.md) for the prior cycle's full scope.

---

## Track A — Anime support

Large active user base; currently underserved by Plex / Emby / Jellyfin (active fork communities exist precisely because the major servers handle anime poorly). The playback side is already solved — intro/credits markers handle OP/ED, trickplay works, audio/subtitle pickers work across direct + HLS. The gap is metadata + scanner work specific to anime conventions.

| Item | Complexity | Notes |
|---|---|---|
| AniDB / AniList / Kitsu metadata agents | M | New providers in `internal/metadata/`, slotted into the existing agent fallback chain (TMDB → TVDB → AniDB → MusicBrainz pattern). AniDB has an HTTP API + UDP API; HTTP is rate-limited (1 req/2 sec) so the agent needs an in-process bucket. AniList has a GraphQL API. Kitsu has REST. Settings UI gets an "anime metadata" section to configure which to prefer. Title-match heuristics for romaji vs kanji vs English title variants. |
| Absolute episode numbering | M | Anime users routinely refer to "Episode 245" rather than S05E12. The scanner must read absolute number from filename (`Show - 245.mkv`) and either store it on the existing `media_items.episode_index` or add a parallel `absolute_episode_index` column. Detail page renders both ("S05E12 · Episode 245"). Episode-list UI offers absolute-number sort. Migration is small; the scanner regex is the work. |
| OVA / ONA / special-episode handling | S | Treat as siblings of regular episodes within the season, with a `kind` field (`episode` / `ova` / `ona` / `special` / `movie`). Detail page groups them under "Specials" instead of mixing with main run. Distinct from the existing `episode` type rather than a new top-level type. |
| Scanlator / release-group tag parsing | XS | Existing Sonarr/TRaSH parser already handles `[group]` markers in folder names; extend to filenames (`[SubsPlease] Show - 01.mkv`). Surfaces in the Settings → Library admin "preferred release group" preference for dedup decisions. |
| Anime-specific content rating | XS | Western TV ratings (`TV-14`, `TV-MA`) don't capture anime nuance; AniList has `R-15` / `R-18` / `R-18+`. Map onto the existing parental-rating-ceiling system. |
| Watching-status metadata mirror | S | "Watching / Plan to Watch / Completed / Dropped / On Hold" is anime-tracker convention (MyAnimeList shape). Could ride the existing favorites + watch-history surfaces or get its own status column. **Open question** — whether to ship as a generic feature (every type benefits) or anime-specific. |

**Dependency order:** metadata agents (foundation — nothing else lands without anime metadata in the DB) → absolute episode numbering → OVA/ONA handling → scanlator parsing → content rating → watching-status.

**Sized total:** ~M + L for the agent + numbering + UX work, rough one-month cycle if nothing else competes.

## Track B — Manga / webtoons

Extends the v2.1 books reader (CBZ / CBR / EPUB) with manga-specific reading conventions. Most of the work is reader-UX flags + metadata; the scanner / storage shape is mostly done.

| Item | Complexity | Notes |
|---|---|---|
| Right-to-left reading order | XS | Reader-side flag on the book; flips page-flip direction and "next page" navigation. Stored as a `reading_direction` field on `media_items` (or in the book's metadata sidecar). MyAnimeList / AniList have this on most manga rows; agent surfaces it during enrichment. |
| Two-page spread mode | S | Reader displays two adjacent pages side-by-side at landscape aspect ratios; collapses to single-page on portrait windows. Spreads handled correctly (a wide cover or splash page that should NOT pair with the next page). Toggle in reader chrome. |
| Vertical-scroll mode (webtoons) | S | Webtoons (Korean / Chinese strip-format comics) have no page breaks — they're one long vertical strip per chapter. Reader mode that ignores page boundaries and renders the full chapter as a scrollable column. Detected by aspect ratio of contained images plus metadata flag. |
| Chapter-aware navigation | S | Manga is read by chapter, not by-volume-by-chapter. Filename / folder conventions (`Vol 03/Chapter 18 - Title.cbz`, or flat `Chapter 18.cbz`) need to be parsed into a chapter tree. The audiobook hierarchy work (`book_author → book_series → audiobook → audiobook_chapter`) gives us a template; manga maps cleanly onto `book_author → book_series → manga_volume → manga_chapter`. |
| MyAnimeList / AniList agent | M | Manga rows are GraphQL-queryable on AniList alongside anime. Same agent infrastructure as Track A — the cost amortises across both tracks. |
| Manga-specific metadata fields | XS | Mangaka (author + artist sometimes split), serialization status (ongoing / completed / hiatus), magazine (Weekly Shōnen Jump etc.), demographic (shōnen / shōjo / seinen / josei) — all available from AniList. Display on the detail page. |

**Dependency order:** chapter-aware navigation (foundation — same shape as audiobook hierarchy) → reading-direction + spread + vertical-scroll modes (parallel — independent reader features) → AniList agent (shared with Track A) → metadata fields.

**Sized total:** ~M, smaller than anime because the books reader and audiobook hierarchy already provide most of the scaffolding.

## Other tracks under consideration

Not yet sized; surface area for the next planning round.

- **TV-client hardware soak** — webOS / Tizen / Roku / Android phone code-complete in v2.1; need real-device validation to flip from ⚠ to ✅ on the comparison matrix.
- **VAAPI hardware encoder validation** — last encoder family pending; needs Linux + non-NVIDIA GPU rig.
- **AMD VCN + Intel QSV decode validation** — encode validation isn't decode validation; pipeline change to set `-hwaccel qsv` / `-hwaccel d3d11va` on those encoder paths and re-run the matrix.
- **iOS + Apple TV apps** — biggest reach gap. Swift skill ramp + App Store review budget. Sized XL — likely deserves its own release line.
- **MCP plugin ecosystem** — the outbound plugin host shipped in v2.1; nothing's been built against it externally. First-plugin announcement post once one exists.

## Items explicitly NOT in v2.2 (and never)

- **Trailers + extras** — per-movie sidecar files (`*-trailer.mp4`, `Trailers/` subfolder) and per-show extras (deleted scenes, behind-the-scenes). Permanently out of scope: the value-vs-effort doesn't pay back on a server that's already covering the playback path well, and "the detail page is empty when there are no extras" is fine.
- **Picture-in-picture in the Tauri desktop shell** — permanently dropped. Tauri 2 webview's missing `requestPictureInPicture` on Windows would force a custom always-on-top miniplayer; not worth the maintenance for a feature most users handle via OS-level windowing instead.

## Items explicitly NOT in v2.2

- **Tidal / Qobuz integration** — sized XL, re-scope decision pending after v2.1 audiophile-pillar feedback. Likely v2.3+ if it survives the re-scope.
- **ML-driven personalised recommendations** — item-to-item collaborative filtering shipped in v2.1 and was pulled because the home-hub row didn't earn its space. No re-attempt without a new product surface that justifies it.
- **Podcast RSS subscription** — out of scope per the project memo (OnScreen doesn't download content; local files work via the regular library scanner).
- **Inbound MCP server** — plugin direction is outbound-only by design (security stance).
- **Game ROMs / emulator integration** — wrong domain. Users layer EmulationStation or LaunchBox on top.
- **IP cameras / surveillance** — wrong domain (Frigate / Blue Iris space).

## Open questions

- **Anime track sizing.** Track A as a whole is ~M+L. Worth scoping down to "metadata agents + absolute numbering" as a v2.2 MVP and deferring OVA/ONA handling + watching-status to v2.3? Or all-in? The MVP gives users the headline differentiator (matching Anime correctly) without the full lift.
- **Manga reader extension vs separate.** Extending the existing `web/src/routes/[book]/+page.svelte` reader with mode flags is the lowest-friction path. Question: does the manga UX deserve a standalone reader (`/manga/[id]`) once the volume / chapter hierarchy lands, or stay as flags on the books reader?
- **Anime watching-status as universal feature.** "Watching / Plan to Watch / Completed / Dropped / On Hold" is anime convention but maps cleanly onto every type (TV shows benefit similarly). Ship as a generic feature on `media_items`, surfaced first in the anime UI, or anime-specific from day one and generalise later?
- **Cadence.** Ship anime + manga together as v2.2 (1.5–2 month cycle), or split — anime in v2.2, manga as v2.3? Splitting halves the MVP risk; bundling lets the AniList agent work amortise across both tracks.
