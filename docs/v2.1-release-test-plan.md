# v2.1 Release Test Plan

The pre-tag QA pass. Anchored against [v2.1-roadmap.md](v2.1-roadmap.md) (everything that landed during v2.1-flight) and the working surface area of v2.0 (everything that should keep working). The plan is structured as a checklist any operator can walk through; each entry names *what* is being verified and *how* to verify it without ambiguity.

Test environment assumptions:
- Dev box: Windows 11 + RTX 5080 + Ryzen 9900X iGPU + Postgres 17 + Memurai/Valkey on 6379. Source of truth for local-build sanity, NVENC + AMF.
- Intel laptop: i9-13900HX (Raptor Lake-HX) — covers QSV (validated 2026-05-01, see comparison-matrix.md).
- Linux production: TrueNAS deploy (RTX 5000) for Linux NVENC + container-shape regression.
- Smart-TV hardware (webOS, Tizen, Roku): pending — covers TV-client paths.
- VAAPI host: pending — needs Linux + non-NVIDIA GPU.

A row's **Status** column is filled in during the test pass: `PASS` / `FAIL <commit-or-issue>` / `SKIP <reason>`. Fail rows block the tag.

## 2. Auth providers (Track D — Playwright spec)

The deepest end-to-end testing in the plan, because v2.0's beta soak found the SAML signing bug in production. Everything here runs against the `docker/docker-compose.idp.yml` stack.

### IdP test stack (one-time setup)

```bash
cd docker
docker compose -f docker-compose.idp.yml up -d
# Verify all three IdPs are reachable:
curl -s http://localhost:8080/health/ready              # Keycloak
ldapsearch -x -H ldap://localhost:389 -b dc=example,dc=org -D "cn=admin,dc=example,dc=org" -w admin
```

Per `docs/auth-test-stack.md`: Keycloak serves both OIDC and SAML realms; OpenLDAP fixture has 3 users (`alice`, `bob`, `carol`) with two groups (`admins`, `users`).

### Automated coverage

Full Playwright suite (136 cases × chromium + firefox + webkit) — **PASS 2026-05-03**: 125 passing, 11 skipped (opt-in rate-limit / WebKit-FLAC gapless / deferred analyser tap / SSE flush placeholder), 0 failing. Run in 2.2 min end-to-end against the live IdP test stack (`docker-compose.idp.yml` brought up earlier).

Per spec file:
- **`smoke.spec.ts` (Tier 1)** — `/health/live` + `/health/ready` (postgres + valkey + migrations all `ok`); web UI loads with title `/OnScreen/i`, zero console errors; 8 libraries from `/api/v1/libraries`; settings page renders TMDB field; UI login + logout round-trip; **API stays responsive (<2s) while a library scan is in flight** (closes the historic DB-pool starvation regression mode).
- **`security.spec.ts` (extended)** — see Section 7 for per-row results. Two regressions found and fixed during the run: trickplay catch-all + media-stream catch-all.
- **`security-tokens.spec.ts`** — per-file stream-token scoping: file A's token returns 401 on file B (file_id claim enforcement) and rejects as Bearer on `/api/v1/*` (purpose claim enforcement). Uses a fresh anonymous request context so the test client's session cookie can't mask a broken token check. Also locks the **fresh-per-call** contract: 3 successive `/items/{id}` requests return 3 distinct PASETO tokens (no server-side caching).
- **`admin.spec.ts` (new this pass)** — `GET /api/v1/admin/backup` returns either a real ZIP (magic `50 4B 03 04`, non-trivial size, `application/zip` content-type) OR a structured 503 with the explicit `PG_DUMP_UNAVAILABLE` error code AND a remediation message (catches the regression mode where the unavailable case silently returns a blank 500). `POST /api/v1/admin/tasks/{id}/run` advances `last_run_at` within 45 s — picks the first known-fast task (DVR retention / EPG refresh / DVR matcher) and polls the task list until the timestamp moves.
- **`sse.spec.ts` (new this pass — currently SKIPPED)** — three parallel `/api/v1/notifications/stream` subscribers should all reach EventSource OPEN. Skipped with TODO: probing from any non-browser client (curl, Node `http`, Playwright EventSource) sits at zero bytes for 30+ seconds even though the route legitimately serves SSE in real browsers (web app SSE is functional in production). Real product behaviour worth its own investigation — likely a Windows dev-build flush ordering issue or a client-header-specific middleware-wrapping bug.
- **`hub.spec.ts`** — v2.1 hub section contract: `continue_watching` + `_tv` + `_movies` + `_other` split, `recently_added`, `recently_added_by_library`, `trending` all present; `cooccurrence` / `because_you_watched` confirmed absent.
- **`transcode.spec.ts` (new this pass)** — DASH removal (no `manifest.mpd` API path, no `manifest_url` in start response, `playlist_url` is present); HLS pipeline smoke (login → library → POST transcode → M3U8 → first segment 200 with `video/MP2T|mp4`); 3 concurrent sessions all return `session_id`; AV1 fMP4 with `video_copy:true` produces `#EXT-X-MAP` + `.m4s` segments (gated on `E2E_AV1_MOVIE_ID`).
- **`playback.spec.ts` (new this pass, chromium-only)** — `/watch/{id}` detail page renders, Play/Resume button visible, `<video>` mounts after click, `readyState ≥ 2`, `currentTime > 0`, seek lands within tolerance, zero console errors.
- **`policy.spec.ts` (new this pass)** — admin marks library `is_private:true`, freshly-registered non-admin user's `/api/v1/libraries` omits it, library is restored to public + test user deleted in finally; home hub contains no "Because you watched" surface.
- **`gapless.spec.ts`** — chromium + firefox: album auto-advance to next track at full volume (gated on `E2E_GAPLESS_ALBUM`). WebKit skipped — can't reliably play 24-bit/192kHz FLAC. Analyser-tap variant deferred (collides with AudioPlayer's ReplayGain `createMediaElementSource` wiring).
- **`auth-providers.spec.ts` (Track D)** — OIDC PKCE shape, SAML signed-AuthnRequest, LDAP end-to-end. Active 2026-05-03 against the live `docker-compose.idp.yml` stack — all 24 cases (8 tests × 3 browsers) PASS. Setup that gets these running is fully script-driven (Keycloak admin REST creates the realm + OIDC client + SAML client + test user; lldap GraphQL + the container's `lldap_set_password` binary creates the LDAP user; OnScreen settings configured via PATCH `/api/v1/settings`). Spec accepts both 302 and 307 for the OIDC redirect (OnScreen handler emits 307 via `http.StatusTemporaryRedirect`; both are valid per RFC 7231).
- **`auth-providers-browser.spec.ts`** — full browser-driven SSO + forgot-password round-trips against the same IdP stack (9 cases, 3 tests × 3 browsers). **OIDC**: clicks the SSO button, fills Keycloak's login form, waits for callback, asserts session cookie set + `/api/v1/users/me/preferences` returns 200 for the JIT-provisioned user. **SAML**: same shape via Keycloak's SAML login → ACS POST round-trip — closes the v2.0 beta-soak SAML signing regression in a real browser, not just at the redirect-shape level. **Forgot-password**: registers a throwaway user with a known email via the admin path, POSTs `/api/v1/auth/forgot-password`, polls Mailpit for the reset email, extracts the token, verifies wrong-token rejection, consumes the real token, logs in with the new password, and asserts replay rejection (single-use enforced). Throwaway user deleted in `finally`. Two setup gotchas surfaced + fixed: (1) Keycloak's `VERIFY_PROFILE` required-action blocked SSO logins because the testuser had no `firstName`/`lastName` — fixed by setting them via Keycloak admin REST PUT. (2) OnScreen SMTP `from` field was empty, causing forgot-password to 400 with "Email is not configured" — fixed via PATCH `/api/v1/settings`.
- **`auth-providers-group-sync.spec.ts` (new this pass)** — four deeper IdP behaviours that each set up their own Keycloak fixtures (group + protocol mappers) so the test is self-contained + idempotent across reruns (12 cases, 4 tests × 3 browsers). **OIDC admin group sync**: adds the `oidc-group-membership-mapper` to the OIDC client, PATCHes OnScreen `oidc.groups_claim` + `admin_group`, walks the SSO flow, asserts the JIT user lands with `is_admin=true`. **SAML admin group sync**: same boundary via three SAML attribute mappers (group-membership + Username + Email — without the latter two the JIT user lands as `G-<uuid>` because OnScreen falls back to the SAML NameID). **OIDC second-login dedup**: walks the SSO flow twice, asserts exactly one user row exists for the username and both flows resolved to the same id. **SAML AuthnRequest single-use**: uses `page.route('**/acs')` to capture the form-POST Keycloak sends to ACS during the legitimate flow, then replays the IDENTICAL body via an anonymous request context — asserts the replay returns 4xx or a 302 to `/login` (RequestTracker single-use rejection). All four tests delete the JIT-provisioned user in `finally` so reruns start clean.

**Caught in this pass — six real bugs that no unit test had exercised:**

1. **NVENC cuda-frame fragility** — `-hwaccel cuda` + scale_cuda + tonemap_cuda/tonemap_opencl chain broke against mainline ffmpeg 8.x + recent NVIDIA drivers across multiple sources (Avengers `No decoder surfaces left`, GoodFellas `-22 EINVAL`). Replaced with uniform software-decode + zscale + GPU-encode shape that matches AMF/QSV. Validated on rows 1, 5, 9, 10 above.
2. **AV1-source-remux into mpegts** — `-c:v copy` of an AV1 source into HLS-mpegts crashed the muxer ("Could not find tag for codec av1"). Switch to fMP4 + `-tag:v av01` for AV1 source remux. Validated on row 7 above. Tests: `TestBuildHLS_AV1Source_Remux_FMP4`, `TestBuildHLS_H264_Remux_StaysMpegTS`.
3. **WebVTT extraction output ordering** — `-c:s webvtt vttPath` was emitted before the HLS positional output, so all the preceding video options accumulated onto the .vtt output and ffmpeg rejected it (`webvtt muxer does not support any stream of type video`). Moved subtitle outputs to AFTER `index.m3u8`. Test: `TestPlayback_Branch_WebVTTExtraction`.
4. **Burn-in Windows path crash** — `subtitleBurnFilter` left backslashes (stripped as escape introducers) and the drive-letter colon (parsed as a key separator) raw, producing `Unable to parse 'original_size' option value 'moviesGoodFellas (1990)…'`. Convert backslashes to forward slashes + escape every colon. Tests: `TestSubtitleBurnFilter_WindowsPath`, `TestPlayback_Branch_SubtitleBurnIn`.
5. **libx265 `-level-idc`** — that's an `hevc_nvenc`-specific option name; libx265 returns `Unrecognized option 'level-idc'`. Removed; libx265 auto-derives level. Test: `TestPlayback_Branch_Encoder_libx265`.
6. **libsvtav1 `-maxrate`** — SVT-AV1 rejects `-maxrate` outside CRF mode (`Max Bitrate only supported with CRF mode`). Skip the maxrate clamp for libsvtav1. Test: `TestPlayback_Branch_Encoder_libsvtav1`.
7. **libx264 inheriting 10-bit from AV1 source** — Chainsaw Man (yuv420p10le AV1) → libx264 emitted 10-bit High 10 H.264, which Chromium (and most browsers) can't decode. Extend the existing `format=yuv420p` strip from AMF/QSV/NVENC to libx264. Test: `TestBuildHLS_Software_StripsTo8Bit`.

### How to repeat the matrix on a different encoder family

Repeating the live matrix on a new host is mechanical:

1. Detect what's available: start the server, grep `bin/server.log` for `transcode encoders`. The `detected:` list tells you which encoders ffmpeg + the GPU advertise on this host.
2. Flip the DB to that encoder family:
   ```sql
   UPDATE server_settings SET value='h264_qsv,hevc_qsv,libx264' WHERE key='transcode_encoders';
   UPDATE server_settings SET value='{"workers":[],"embedded_enabled":true,"embedded_encoder":"h264_qsv,hevc_qsv,libx264"}'::jsonb WHERE key='worker_fleet';
   ```
   Adjust the encoder names per host (`h264_amf,hevc_amf,libx264` / `h264_vaapi,hevc_vaapi,libx264` / etc.).
3. Restart the server. Confirm the startup log line shows `active:` matching what you set.
4. Run the matrix script. The 11 rows are parameterized by item UUID + request body — the script lives in shell history (search for `=== AMF live path matrix ===` in the conversation log) but the simplest reproduction is curl-by-hand against each row in the table above.
5. After: `taskkill //F //IM ffmpeg.exe` (Windows) / `pkill -f ffmpeg` (Linux/Mac) to clean up orphaned encoder processes — the test scripts POST `/transcode` + GET `/playlist.m3u8` but never POST `/stop`, so ffmpeg keeps running until the worker's heartbeat-loss timeout fires (~30s after the server is killed).

The script that wrapped the 11 calls into one shell function (`run_test "<label>" "<item-uuid>" '<json-body>' <playlist-timeout-s>`) is reproduced verbatim in the conversation log; copy from there if you want to re-run the full set rather than spot-checking individual rows.
