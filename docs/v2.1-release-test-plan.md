# v2.1 Release Test Plan

The pre-tag QA pass. Anchored against [v2.1-roadmap.md](v2.1-roadmap.md) (everything that landed during v2.1-flight) and the working surface area of v2.0 (everything that should keep working). The plan is structured as a checklist any operator can walk through; each entry names *what* is being verified and *how* to verify it without ambiguity.

Test environment assumptions:
- Dev box: Windows 11 + RTX 5080 + Ryzen 9900X iGPU + Postgres 17 + Memurai/Valkey on 6379. Source of truth for local-build sanity, NVENC + AMF.
- Intel laptop: i9-13900HX (Raptor Lake-HX) — covers QSV (validated 2026-05-01, see comparison-matrix.md).
- Linux production: TrueNAS deploy (RTX 5000) for Linux NVENC + container-shape regression.
- Smart-TV hardware (webOS, Tizen, Roku): pending — covers TV-client paths.
- VAAPI host: pending — needs Linux + non-NVIDIA GPU.

A row's **Status** column is filled in during the test pass: `PASS` / `FAIL <commit-or-issue>` / `SKIP <reason>`. Fail rows block the tag.

## 2. Auth providers (Track D — Playwright spec)

The deepest end-to-end testing in the plan, because v2.0's beta soak found the SAML signing bug in production. Everything here runs against the `docker/docker-compose.idp.yml` stack.

### IdP test stack (one-time setup)

```bash
cd docker
docker compose -f docker-compose.idp.yml up -d
# Verify all three IdPs are reachable:
curl -s http://localhost:8080/health/ready              # Keycloak
ldapsearch -x -H ldap://localhost:389 -b dc=example,dc=org -D "cn=admin,dc=example,dc=org" -w admin
```

Per `docs/auth-test-stack.md`: Keycloak serves both OIDC and SAML realms; OpenLDAP fixture has 3 users (`alice`, `bob`, `carol`) with two groups (`admins`, `users`).

### Per-provider tests

| What | How | Status |
|---|---|---|
| **Local** — register | Open `/setup` on a fresh DB, create admin via the wizard. Confirm Postgres has the user row and a session cookie is set. | |
| **Local** — login | Logout, hit `/login`, submit credentials. Confirm redirect to home + cookie. | |
| **Local** — password reset | Trigger via `/auth/forgot`, capture email from `bin/server.log` (dev SMTP catcher), follow link, set new password, log in with it. | |
| **OIDC** — client setup | In `/admin/auth`, configure issuer URL `http://localhost:8080/realms/onscreen`, client_id, secret. Save → reload → confirm config persisted. | |
| **OIDC** — login flow | Hit `/login` → "Sign in with OIDC" → Keycloak login form → consent → redirect back to OnScreen. New user row created via JIT, session cookie set. | |
| **OIDC** — claim mapping | Confirm user.email + user.username + admin-group sync match the IdP claims (group `admins` in Keycloak = `is_admin=true`). | |
| **OIDC** — second login | Same OIDC user logs in again. Confirm no duplicate row, session refreshed. | |
| **SAML** — metadata import | Upload Keycloak's IdP metadata XML in `/admin/auth`. Confirm SP metadata becomes available at `/auth/saml/metadata`. | |
| **SAML** — login flow | Hit `/login` → "Sign in with SAML" → Keycloak SAML login → consent → redirect to ACS → session set. **Critical — this is the path the v2.0 beta soak broke**: confirm AuthnRequest is signed (look in `bin/server.log` for the signing line), confirm RelayState round-trip, confirm response signature validation. | |
| **SAML** — admin group sync | User in Keycloak's `admins` group lands as `is_admin=true` in the OnScreen DB after first login. | |
| **SAML** — single-use AuthnRequest | Replay the same AuthnRequest at the ACS (capture from browser, replay via `curl`). Confirm 4xx — RequestTracker rejects the replay. | |
| **SAML** — Valkey-backed tracker | Restart the OnScreen server mid-flow (after AuthnRequest, before ACS). Confirm the tracker survives the restart (Valkey-backed) and the ACS still validates. | |
| **LDAP** — bind | Configure LDAP in `/admin/auth` with bind DN + search filter. Test connection button reports success. | |
| **LDAP** — login | `/login` → "Sign in with LDAP" → submit `alice` / fixture password → session cookie set, user JIT-created. | |
| **LDAP** — group sync | `bob` is in fixture group `admins` → after login, `is_admin=true`. `carol` (regular user) → `is_admin=false`. | |
| **Magic link** | `/auth/magic` flow with the dev SMTP catcher capturing the email. Single-use enforced (second click on the link 4xxs). | |
| **Cross-cutting** — session cookies | Login (any provider). Confirm cookies have `HttpOnly; Secure; SameSite=Lax`; not visible to JS. | |
| **Cross-cutting** — refresh token rotation | Wait for access token to expire (or shorten TTL via env), hit any authenticated endpoint, confirm a fresh access token is minted via the rotated refresh token. | |
| **Cross-cutting** — logout | `/logout` clears cookies + revokes session in Valkey. Subsequent request to authed endpoint returns 401. | |
| **Per-user policy** — auto-grant template | Set "auto-grant new users to library X" in admin. Create new user via OIDC. Confirm they can see library X but not Y on first login. | |
| **Per-user policy** — `is_private` flag | Mark library Y as `is_private=true`. Confirm users without explicit grant don't see it in `/api/v1/libraries`. | |
| **Per-user policy** — content-rating filter | Set parental rating ceiling on a managed profile to PG-13. Confirm `/api/v1/hub` and search exclude R-rated items. | |
| **Per-user policy** — view-as | Admin "view as" target user. Confirm rendered hub matches what target sees on their own login (read-only, no GET-mutation). | |

### Automated coverage

The Playwright spec at `web/tests/e2e/auth-providers.spec.ts` (per Track D) covers: OIDC PKCE shape, SAML signed-AuthnRequest, LDAP login. Run before the manual pass:

```bash
cd web
npm run test:e2e -- auth-providers
```

A green Playwright run is necessary but not sufficient — the manual matrix above catches IdP-config UI regressions the spec doesn't exercise.

## 7. Security regression

| What | How |
|---|---|
| Rate limiting | Hit `/api/v1/login` with bad credentials 20 times in 30s. Confirm 429 after the configured threshold. |
| JWT expiration | Modify token expiry to 60s, wait 65s, hit any authed endpoint. Confirm 401 + auto-refresh attempt via cookie. |
| CSRF | POST to a state-changing endpoint without the CSRF token (or via cross-origin fetch). Confirm 403. |
| XSS in user-controlled fields | Set library name to `<script>alert(1)</script>`, view in admin UI. Confirm renders as text, not execution. |
| Path traversal in segment URLs | `GET /api/v1/transcode/sessions/abc/seg/..%2Fpasswd?token=t` — confirm 4xx, not the file. |
| Per-file stream token scoping | Issue a token for file A, attempt to use it for file B — confirm 401. Confirm 24h expiry. |
| OIDC state replay | Capture a successful OIDC callback URL, replay it — confirm 4xx (state is single-use). |
| Admin-endpoint authorization | Login as non-admin, attempt to hit `/admin/users` — confirm 403. |

## Tag-blocking criteria

Tag v2.1 only when:
- All `Status` cells in sections 1, 2, 3 are PASS or explicitly waived.
- All four IdP rows in section 2 PASS (the SAML row is the soak-found regression we cannot ship broken).
- Encoder vendor matrix has at least: NVENC + AMF + QSV + libx264 + libsvtav1 PASS. VAAPI may stay ⚠️ if Linux + non-NVIDIA hardware unavailable, with a note in the release announcement.
- Async OCR (Track A) PASS against the live beta site.
- DASH-removal regression (Track H "is DASH gone everywhere") PASS.

Document any waived rows in the release notes' "known limitations" section.
